## 建构方观点

- **核心方案**: 采用分层记忆架构 - 工作记忆(会话级) + 情景记忆(对话历史) + 语义记忆(知识库) + 程序记忆(技能/偏好)

- **存储技术栈**: 
  - 向量数据库用于语义检索
  - 结构化存储用于元数据和关系
  - 本地文件系统用于轻量场景(JSON/Markdown)

- **记忆压缩策略**: 
  - 定期执行摘要压缩，避免线性增长
  - 关键信息提取 + 时间衰减权重
  - 重要性评分机制自动筛选持久化内容

- **检索增强**: 
  - 混合检索(向量相似度 + 关键词 + 时间窗口)
  - 上下文感知的记忆召回
  - 用户意图触发的主动记忆关联

- **行动建议**: 
  1. 优先实现"会话摘要 + 向量存储"的最小可行方案
  2. 设计 Memory Manager 统一接口，支持多后端切换
  3. 建立记忆重要性评分模型，实现智能筛选
  4. 添加用户可编辑的记忆入口(如 CLAUDE.md 模式)

## 存储专家 视角

- **向量数据库 + 关系库混合架构**：短期记忆用 Redis/内存，长期记忆用 Milvus/Pinecone 做语义检索，元数据存 PostgreSQL 保证 ACID

- **记忆分层冷热分离**：高频访问记忆驻留内存，中频下沉向量库，低频归档对象存储，自动 TTL 淘汰无效记忆

- **写入优化建议**：采用 Write-Ahead Log 先写本地再异步同步，避免阻塞 Agent 推理；记忆去重用 SimHash + 阈值过滤

### 致命问题
1. 记忆膨胀问题：无限增长的上下文如何处理？Token 成本和响应延迟会指数级上升，任何方案是否都难逃这个物理限制？

### 潜在风险
- 隐私合规：持久化的记忆数据涉及用户敏感信息，GDPR/个保法合规成本高昂
- 记忆污染：错误记忆一旦持久化，如何修正？可能造成 Agent 行为永久偏离
- 存储依赖：外部存储单点故障会导致 Agent "失忆"，可用性与一致性难以兼得
- 跨会话上下文断裂：不同对话场景的记忆如何隔离？记忆混淆会导致 Agent 产生幻觉

  - 阈值触发压缩，保持存储可控
